## 1. Architecture Decision Record

### ADR-001: Tauri over Electron
**Decision:** Use Tauri (Rust backend + WebView2/WebKit frontend)  
**Rationale:**
- 10–15x lower memory footprint than Electron (critical for a graph-heavy app)
- Native OS performance for file I/O and IPC
- Smaller binary distribution (~10MB vs ~150MB for Electron)
- Rust backend enables safe, performant local data operations
- WebView2 on Windows gives us a modern Chromium engine without bundling it
**Trade-off:** Rust learning curve for backend; smaller ecosystem than Electron. Mitigated by keeping Rust surface small — only IPC, file system, and HTTP calls.

### ADR-002: Pixi.js for Skill Tree Graph Rendering
**Decision:** Use Pixi.js (WebGL-accelerated 2D renderer) for the skill tree graph  
**Rationale:**
- WebGL-backed — handles 200+ node trees at 60fps without issue
- Battle-tested for graph-heavy tools (used in similar game companion apps)
- React integration via `@pixi/react` or manual canvas mounting
- Supports zoom/pan natively via container transforms
- Fallback to Canvas2D if WebGL unavailable
**Trade-off:** More setup than SVG/D3. Mitigated by the fact that SVG at 100+ nodes becomes a performance bottleneck.

### ADR-003: React + TypeScript Frontend
**Decision:** React 18 with TypeScript as the UI framework  
**Rationale:**
- Strong ecosystem for complex state management (Zustand for app state)
- TypeScript catches data model mismatches early — critical when modeling complex game data
- Excellent Tauri integration via `@tauri-apps/api`
- Component model aligns well with panel-based UI architecture

### ADR-004: Deterministic Scoring Engine (not AI-driven)
**Decision:** Build scores (0–100 per dimension) are computed by a deterministic scoring engine, NOT generated by the AI  
**Rationale:**
- Consistency: same build always gets same score
- Speed: scores update in real-time as user clicks nodes (no API round-trip)
- Cost: avoids Claude API call per node click
- Trust: users can reason about the score formula
**AI role:** Claude generates *suggestion lists* and *explanation text* only. The scoring engine validates and quantifies the suggestions.

### ADR-005: Claude API with Structured Output
**Decision:** Use Claude (claude-sonnet-4-6) with structured JSON output mode for optimization suggestions  
**Rationale:**
- Structured output guarantees parseable suggestion format
- Streaming enables progressive UI updates during optimization
- claude-sonnet-4-6 balances intelligence and speed for this use case
- Prompt caching on the game data context block reduces cost significantly

### ADR-006: SQLite for Local Data Storage
**Decision:** SQLite (via Tauri's `tauri-plugin-sql`) for local build storage and game data cache  
**Rationale:**
- Relational model fits the game data structure (classes → masteries → trees → nodes)
- ACID guarantees for build saves
- Zero external dependencies — file-based, ships with the app
- Fast queries for node lookups during scoring

### 3.3 Skill Tree Graph (Pixi.js)

The graph renderer is isolated as a self-contained module that receives data via props and emits events.

```typescript
interface SkillTreeGraphProps {
  tree: PassiveTree | SkillTree;
  allocations: Map<string, number>;
  suggestions: Suggestion[];
  onNodeClick: (nodeId: string) => void;
  onNodeHover: (nodeId: string | null) => void;
}
```

**Rendering pipeline:**
1. Parse tree nodes + edges → build adjacency graph
2. Layout: use stored x/y coordinates from game data (Last Epoch trees have fixed layouts)
3. Render edges first (z-index below nodes)
4. Render nodes as circles with fill color based on state
5. Render suggestion glow rings as separate layer above nodes
6. Mount tooltip layer as DOM overlay (not Pixi — for accessibility + HTML rendering)

**Zoom/pan:**
- Implemented via Pixi `Container.scale` and `Container.position`
- Wheel event: scale ±0.1 per tick, clamped [0.3, 2.5]
- Drag: pointer events on stage

## 4. Scoring Engine

### 4.1 Design

The scoring engine is a pure TypeScript module — no external dependencies, no API calls. It takes a `Build` and returns `{ damage: number, survivability: number, speed: number }` on a 0–100 scale.

```typescript
interface ScoringEngine {
  score(build: Build, gameData: GameDataStore): BuildScores;
}
```

### 4.2 Scoring Model

Each dimension has a **weighted tag accumulator**:

**Damage score:**
- Sum weighted values of allocated nodes with damage-relevant tags
- Tags: `increased_damage`, `critical_strike`, `penetration`, `flat_damage`, `multiplier`
- Weight by tag importance (multipliers > penetration > flat > increased)
- Normalize to 0–100 against the theoretical maximum for that mastery

**Survivability score:**
- Tags: `increased_health`, `increased_armor`, `damage_reduction`, `dodge`, `block`, `resist`, `leech`
- Weight armor/dodge/resist equally; leech scores lower (active mitigation preferred)
- Normalize same way

**Speed score:**
- Tags: `movement_speed`, `attack_speed`, `cast_speed`, `cooldown_recovery`
- Movement speed weights highest

**Normalization:**
- `max_possible` per mastery computed at game data load time (all nodes allocated hypothetically)
- `score = (player_total / mastery_max) * 100`, clamped 0–100

### 4.3 Tag Data

Node tags are extracted from community game data. Each node has:
```typescript
interface Node {
  id: string;
  name: string;
  description: string;
  x: number; y: number;
  maxPoints: number;
  connections: string[]; // node IDs
  tags: NodeTag[];
  effects: NodeEffect[]; // parsed numeric values
}

interface NodeEffect {
  stat: StatType;
  value: number;
  type: 'flat' | 'increased' | 'more';
}
```

## 5. AI Integration (Claude API)

### 5.1 Optimization Flow

```
User clicks "Optimize"
  → Frontend dispatches OptimizationStore.runOptimization()
  → Tauri IPC: invoke('optimize_build', { buildState, gameDataContext, goal })
  → Rust handler: construct prompt → call Claude API (streaming)
  → Stream chunks back to frontend via Tauri event emitter
  → Frontend: parse streaming JSON → update suggestions list progressively
  → On complete: validate all suggestions against game data → filter invalid
```

### 5.2 Prompt Architecture

**System prompt (static — cached):**
```
You are an expert Last Epoch build optimizer with deep knowledge of game mechanics, 
damage formulas, and build theory. You analyze skill tree allocations and provide 
specific, actionable optimization suggestions.

You always respond in valid JSON matching the OptimizationResponse schema.
You never suggest allocating nodes that are not connected to the player's 
existing allocation path.
Your explanations are technical and precise — users are experienced min-maxers.
```

**User prompt template:**
```
OPTIMIZATION GOAL: {goal}

CURRENT BUILD:
Class: {className}
Mastery: {masteryName}
Active Skills: {skillList}
Points Used: {used}/{total}

ALLOCATED NODES:
{allocatedNodeList with descriptions}

UNALLOCATED REACHABLE NODES:
{reachableNodeList with descriptions}

CURRENT SCORES:
Damage: {score.damage}/100
Survivability: {score.survivability}/100  
Speed: {score.speed}/100

Generate up to 5 optimization suggestions. For each suggestion, specify:
1. The node to add, remove, or swap
2. The reason it improves the stated goal
3. Estimated score delta per dimension

Respond in this JSON schema:
{OptimizationResponse schema}
```

**Prompt caching strategy:**
- System prompt: always cache (static)
- Game data context block (node descriptions): cache with `cache_control: ephemeral` — reused across multiple requests in a session
- Build state: never cached (changes per request)

### 5.3 Response Schema

```typescript
interface OptimizationResponse {
  suggestions: Array<{
    type: 'add' | 'remove' | 'swap';
    nodeId: string;
    swapNodeId?: string; // for 'swap' type
    scoreDelta: {
      damage: number;    // -10 to +10
      survivability: number;
      speed: number;
    };
    explanation: string; // 2-4 sentences, technical
    rank: number;        // 1 = highest impact
  }>;
  overallAnalysis: string; // 1-2 sentence summary of the build's current state
}
```

### 5.4 Post-Processing (Validation Layer)

Before suggestions reach the UI, the frontend validates each suggestion:
1. `nodeId` exists in game data ✓
2. For `add`: node is reachable from current allocations ✓
3. For `remove`: node can be removed without disconnecting other allocated nodes ✓
4. For `swap`: both nodes valid ✓
5. Invalid suggestions are filtered out silently (logged to console)

## 6. Data Pipeline

### 6.1 Game Data Fetch

**Source:** lastepochtools.com community API (or scraped wiki data)  
**Trigger:** App launch (if cache is stale or absent)  
**Storage:** SQLite local database

**Schema:**
```sql
CREATE TABLE classes (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT
);

CREATE TABLE masteries (
  id TEXT PRIMARY KEY,
  class_id TEXT REFERENCES classes(id),
  name TEXT NOT NULL,
  description TEXT
);

CREATE TABLE passive_nodes (
  id TEXT PRIMARY KEY,
  mastery_id TEXT REFERENCES masteries(id),
  name TEXT NOT NULL,
  description TEXT NOT NULL,
  x REAL NOT NULL,
  y REAL NOT NULL,
  max_points INTEGER NOT NULL DEFAULT 1,
  tags TEXT NOT NULL,  -- JSON array
  effects TEXT NOT NULL -- JSON array of NodeEffect
);

CREATE TABLE passive_edges (
  from_node_id TEXT REFERENCES passive_nodes(id),
  to_node_id TEXT REFERENCES passive_nodes(id),
  PRIMARY KEY (from_node_id, to_node_id)
);

CREATE TABLE skills (
  id TEXT PRIMARY KEY,
  mastery_id TEXT REFERENCES masteries(id),
  name TEXT NOT NULL,
  description TEXT,
  damage_types TEXT -- JSON array
);

CREATE TABLE skill_nodes (
  id TEXT PRIMARY KEY,
  skill_id TEXT REFERENCES skills(id),
  name TEXT NOT NULL,
  description TEXT NOT NULL,
  x REAL, y REAL,
  max_points INTEGER NOT NULL DEFAULT 1,
  tags TEXT NOT NULL,
  effects TEXT NOT NULL
);

CREATE TABLE skill_edges (
  from_node_id TEXT REFERENCES skill_nodes(id),
  to_node_id TEXT REFERENCES skill_nodes(id),
  PRIMARY KEY (from_node_id, to_node_id)
);

CREATE TABLE data_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- key='last_fetched', value=ISO timestamp
-- key='game_version', value='1.x.x'

-- Saved builds
CREATE TABLE builds (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  class_id TEXT NOT NULL,
  mastery_id TEXT NOT NULL,
  passive_allocations TEXT NOT NULL, -- JSON
  skill_allocations TEXT NOT NULL,   -- JSON
  equipped_skills TEXT NOT NULL,     -- JSON
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

### 6.2 Cache Invalidation

- On launch: compare stored `game_version` against latest from community API
- If version mismatch or cache > 7 days old: re-fetch
- User can manually trigger refresh via Settings

### 6.3 Build Import

**Format:** lastepochtools.com build URL  
**Process:**
1. Extract build ID from URL
2. GET `https://www.lastepochtools.com/api/builds/{id}`
3. Map response to internal `Build` schema
4. Validate all referenced nodes exist in local game data
5. Populate build store

### 7.2 Streaming (Claude API → Frontend)

The optimization command uses Tauri's event system to stream Claude's response:

```rust
// Emit chunks as they arrive from Claude streaming API
window.emit("optimization-chunk", chunk)?;
window.emit("optimization-complete", ())?;
window.emit("optimization-error", error_message)?;
```

Frontend listens:
```typescript
await listen<string>('optimization-chunk', (event) => {
  optimizationStore.appendChunk(event.payload);
});
```

### 7.3 API Key Storage

Stored via Tauri's `keyring` plugin (OS keychain: Windows Credential Manager / macOS Keychain).  
Never written to plain text files.

## 10. Security Considerations

- **API key:** Stored in OS keychain only — never in plaintext, never sent to any server except Anthropic
- **No user data leaves the device** except the build state sent to Claude API for optimization (and only when user explicitly clicks "Optimize")
- **Build data:** Stays local in SQLite — no server sync in MVP
- **Community data fetch:** HTTPS only; validate response schema before storing
- **Input validation:** All community API responses are schema-validated before SQLite insertion (prevent injection via malformed game data)

## 11. Performance Targets & Strategies

| Target | Strategy |
|--------|----------|
| Tree render < 500ms | Pixi.js WebGL; nodes batch-rendered as sprites |
| Node click < 50ms | Pixi interaction events are synchronous; state update is O(1) |
| Score recalc < 16ms | Pure TS scoring engine; runs on every node click; memoized per allocation hash |
| AI response < 10s | Streaming + prompt caching; game data context cached with Anthropic |
| App launch < 3s | SQLite data served instantly from disk; game data pre-fetched and cached |
